### 1.4 data.go.kr의 3가지 제공 패턴

data.go.kr이 모든 API의 게이트웨이 역할을 하는 것이 아니다. 실제로는 3가지 패턴이 혼재하며, 이 사실을 모르는 개발자들이 자주 혼동한다.

| 패턴 | 설명 | data.go.kr serviceKey로 호출 | 별도 가입 |
|------|------|:---:|:---:|
| **A: 완전 게이트웨이** | `apis.data.go.kr` 도메인으로 직접 호출 가능 | O | X |
| **B: 인덱스(카탈로그)만** | 목록만 제공, "활용신청" 시 원본 사이트로 이동 | **X** | **O** |
| **C: 양쪽 제공 (범위 차이)** | 양쪽에 API가 있지만 버전/범위/키가 다름 | 부분적 | O (전체 기능 시) |

#### 패턴별 해당 기관

**Pattern A (게이트웨이 — data.go.kr 키만으로 OK):**
- 국토교통부 실거래가, 소상공인 상가정보, 국세청 사업자등록, 행정안전부 인구통계, 에어코리아 대기정보

**Pattern B (인덱스만 — 원본 사이트 별도 가입 필수):**
- KIPRIS(특허), 한국도로공사, 오피넷(유가), 워크넷(채용), 국립중앙도서관, DART(전자공시), 서울열린데이터, 경기데이터드림

**Pattern C (양쪽 제공, 범위 차이 — 가장 혼동 유발):**

| 기관 | data.go.kr 버전 | 원본 사이트 | 차이점 |
|------|----------------|-----------|--------|
| **TourAPI (관광공사)** | 구버전 유지 | api.visitkorea.or.kr (4.0) | data.go.kr은 업데이트 느림, 최신 기능 미반영 |
| **기상청** | 기본 예보만 | apihub.kma.go.kr | API허브에 레이더, 위성, 특보 등 추가 API |
| **도로명주소** | 기본 검색만 | business.juso.go.kr | **키 체계 다름** (serviceKey vs confmKey) |
| **건축물대장** | 표제부/총괄표제부 | open.eais.go.kr | 세움터에 인허가 등 추가 API |
| **식품안전나라** | 극히 일부 | openapi.foodsafetykorea.go.kr | 원본이 **주요 채널**, 자체키 필요 |
| **건보심사평가원** | 병원/약국 기본 | opendata.hira.or.kr | 원본에 상세 의료통계 추가 |

#### 개발자가 가장 많이 겪는 혼동

1. **키 호환 불가**: data.go.kr `serviceKey`로 원본 사이트 API를 호출하면 **인증 실패**. 도로명주소(`confmKey`), 식품안전나라, KIPRIS 등에서 빈번.
2. **버전 불일치**: data.go.kr API 문서가 구버전 기준. 블로그 코드를 복사하면 파라미터가 안 맞음.
3. **검색 함정**: data.go.kr에서 검색하면 **파일 데이터(CSV)**와 **실시간 API**가 섞여 나옴. 같은 데이터가 여러 건 중복 등록.
4. **호출 한도 차이**: data.go.kr 경유 시 일 1,000회 제한이 걸리지만, 원본 사이트 직접 발급은 더 높은 한도 제공.

> **핵심**: data.go.kr ID 1개로 "70% 커버 가능"이라는 이전 분석은 **Pattern A에만 해당**. Pattern B/C까지 포함하면 실제 커버리지는 **40% 이하**이며, 나머지는 원본 사이트 별도 가입이 필요하거나 data.go.kr 버전이 제한적이다.

### 2.3 주요 문제점

#### (1) 표준화 부재 — 포털마다 모든 것이 다르다

| 차원 | data.go.kr | 서울 | ECOS | DART | KOSIS |
|------|-----------|------|------|------|-------|
| 인증키 파라미터명 | `serviceKey` | URL path | URL path | `crtfc_key` | `apiKey` |
| 인증키 전달 위치 | Query param | Path | Path | Query param | Query param |
| 페이징 방식 | pageNo+numOfRows | 시작/끝 인덱스 | 시작/끝 인덱스 | page_no+page_count | 기간 기반 |
| 에러코드 체계 | resultCode (00~99) | CODE (INFO-xxx) | 자체 | status (0xx) | 자체 |

> 범용 API 클라이언트를 만들 수 없다. 포털마다 별도 어댑터가 필요하다.

#### (2) 에러 시에도 항상 HTTP 200 반환

모든 한국 공공 API가 공유하는 가장 심각한 문제. 인증 실패, 파라미터 오류, 호출 초과 모두 HTTP 200 OK를 반환하고, body 안에 자체 에러코드를 넣는다.

```
HTTP/1.1 200 OK     ← 실제로는 인증 실패인데 200
Content-Type: application/xml

<response>
  <header>
    <resultCode>30</resultCode>
    <resultMsg>SERVICE_KEY_IS_NOT_REGISTERED_ERROR</resultMsg>
  </header>
</response>
```

#### (3) serviceKey 이중 인코딩 — 개발자 통과의례

data.go.kr 인증키에 `+`, `/`, `=` 등 Base64 특수문자가 포함되어 있어, HTTP 클라이언트의 자동 URL 인코딩과 충돌. 관련 블로그 글이 수백 개에 달하는 악명 높은 문제.

#### (4) CORS 미지원 — 프론트엔드 직접 호출 불가

대부분의 공공 API가 CORS 헤더를 설정하지 않아, 브라우저 JavaScript에서 직접 호출이 불가능하다. 반드시 백엔드 프록시 서버가 필요하다.

#### (5) API Key 2년 만료, 수동 갱신

data.go.kr 인증키는 2년 유효기간이 있으며, **자동 갱신이 되지 않는다**. 만료를 놓치면 서비스가 중단되며, 이는 실제 운영 서비스에서 매우 흔한 장애 원인이다.

#### (6) Rate Limit 헤더 없음

`X-RateLimit-Limit`, `X-RateLimit-Remaining` 같은 표준 헤더가 없어, 남은 호출 가능 횟수를 사전에 알 수 없다. 초과해야 비로소 에러로 알 수 있다.

### 3.3 AI Agent가 공공 API를 쓸 때 겪는 문제

```
[AI Agent 시나리오]

1. API 스펙 파악: OpenAPI 스펙이 없어 → 사람이 수동으로 function schema 작성 필요
2. API Key 발급: 사람이 직접 포털 가입 + 신청 → AI가 자동 수행 불가
3. API 호출: 포털마다 URL 패턴, 파라미터명, 인코딩 규칙이 달라 → 포털별 어댑터 수동 구현
4. 응답 파싱: XML 기본이라 토큰 2.5배 소모 → 비용 증가
5. 에러 처리: HTTP 200인데 body에 에러 → 자동 판단 불가, 포털별 에러코드 학습 필요
6. Rate 관리: 남은 호출량 모름 → 보수적 호출 or 갑자기 차단
7. Key 갱신: 2년 후 만료 통보 → 사람이 수동 갱신
```

| 샌드박스 | **2** | 7 | -5 |
| **종합** | **2.0/10** | **7.4/10** | **-5.4** |

### 3.7 주목할 해외 사례

**미국 GovInfo MCP 서버** — GPO(정부인쇄국)가 Web UI, REST API, **MCP**의 3채널로 연방 문서를 제공. AI agent가 법률/정책 문서를 직접 검색·분석 가능.

**프랑스 data.gouv.fr MCP** — 정부 공식 MCP 서버를 공개 운영 (접근 제한 없음). AI 챗봇이 대화형으로 데이터셋 검색·탐색·분석 가능.

**영국 OAuth 2.0 의무화** — API Key를 인증 수단으로 **명시적 금지**. "쉽게 가로채고 재사용할 수 있다"는 이유. 한국은 아직 URL에 키를 노출하는 방식.

**싱가포르 Key-Free 테스트** — API Key 없이도 브라우저에서 바로 테스트 가능. 가입 전에 API를 체험할 수 있는 개발자 친화적 설계.

## 4. 공동인증서 기반 API

### 4.1 현황 분류

| 구분 | 서비스 | AI 자동화 가능성 |
|------|--------|----------------|
| **API 호출에 전자서명 필수** | 국세청 전자세금계산서, 홈택스 내부 프로토콜, 마이데이터 통합인증 | **불가** (사용자 대화형 인증 전제) |

| **서드파티가 대행** | CODEF, 하이픈, 팝빌 | **반자동** (등록/갱신은 수동, 이후 REST API) |

### 4.2 서드파티 프록시 플랫폼

| 플랫폼 | 방식 | 커버리지 | 비용 |
|--------|------|---------|------|
| CODEF (헥토데이터) | 인증서 파일 → Connected ID → REST API | 홈택스, 건보, 국민연금, 금융기관 | 유료 |
| 하이픈 | OAuth 2.0 + 내부 스크래핑 | 500+ API (홈택스, 건보 등) | 유료 |
| 팝빌 | 인증서 등록 → HMAC-SHA256 토큰 | 전자세금계산서, 홈택스 수집 | 유료 |

> **구조적 한계**: 공동인증서는 1년 주기 갱신이 필요하며, 갱신은 반드시 사람이 수행해야 한다. 완전 무인 자동화는 현실적으로 불가능하다.

## 6. 핵심 문제점 요약

### 6.1 구조적 문제

| 문제 | 원인 | 영향 |
|------|------|------|
| **60+ 포털 파편화** | 기관별 독자 운영, 통합 거버넌스 부재 | 개발자가 30~40개 ID 필요, 포털마다 학습 필요 |
| **2010년대 설계 동결** | XML/SOAP 시대에 설계 후 근본 개선 없음 | JSON 시대/AI 시대에 부적합 |
| **공급자(기관) 중심 설계** | "데이터 제공"에 초점, "개발자 경험"은 부차적 | 가입/승인/문서/에러 모두 불편 |
| **HTTP 의미론 무시** | 에러도 200 OK, 상태코드 미활용 | 자동화/AI 에러 처리 불가 |
| **인증 모델 구식** | URL에 키 노출, 공동인증서 레거시 | 보안 취약, AI 자동화 제한 |

### 6.2 운영적 문제

| 문제 | 내용 |
|------|------|
| API Key 2년 만료 + 수동 갱신 | 장기 서비스 운영 시 인적 개입 불가피 |
| 사전 공지 없는 장애/점검 | 안정성 보장 불가, 자체 모니터링 필수 |
| API 폐지/스펙 변경 무공지 | 운영 중 갑자기 작동 중단 가능 |
| 트래픽 증량 승인 3일~2주 | 서비스 확장 시 병목 |
| 공공누리 유형별 라이선스 상이 | API마다 개별 확인 필요 |

### 6.3 AI 시대 부적합 문제

| 문제 | AI에 미치는 영향 |
|------|----------------|
| OpenAPI 스펙 미제공 | LLM function calling에 수동 스키마 작성 필수 |
| XML 기본 응답 | 같은 데이터에 JSON 대비 2.5배 토큰 소모 → 비용 증가 |
| HTTP 200 일관 반환 | AI agent 자동 에러 판단 불가 |
| Rate Limit 헤더 없음 | AI 자동 속도 조절 불가 |
| CORS 미지원 | 웹 기반 AI 앱 직접 연동 불가 |
| 포털별 비표준 패턴 | MCP 서버를 포털마다 개별 구현해야 함 |

### 7.5 가장 시급한 단일 개선

> **OpenAPI 3.0 스펙 파일 의무 제공**
>

> - MCP 서버를 자동 생성할 수 있다
> - LLM function calling tool로 바로 등록할 수 있다
> - API 문서를 기계가 읽을 수 있다
> - 클라이언트 SDK를 자동 생성할 수 있다
> - AI 시대 적합성이 극적으로 향상된다
>
> 현재는 60개 포털의 수천 개 API 스펙을 사람이 일일이 읽고 수동으로 스키마를 작성해야 한다. 이것이 공공 데이터 AI 활용의 가장 큰 병목이다.
>
> **참고**: 영국은 이미 OpenAPI를 정부 API 필수 표준으로 지정했고, EU는 전 API에 적용 완료했다.

## 10. 결론

대한민국 공공 Open API 생태계는 **"데이터 공개"에는 성공했지만, "데이터 활용 용이성"에서는 2010년대 수준에 머물러 있다.**

### 핵심 수치

| 지표 | 값 |
|------|-----|
| 총 포털 수 | **169개** |
| 추정 API 수 | **~19,198개** |
| 전체 커버 시 필요 ID | **130개** |
| data.go.kr만으로 완전 동작 | **18%** (31개 포털) |
| 사용자 등급 A~B | **8개** (4.7%) |
| AI 등급 A~B | **2개** (1.2%) |
| OECD 공공데이터 개방 지수 | **1~2위** |
| AI 적합성 점수 | **2.0/10** (해외 7.4) |

### 핵심 메시지

- **사용자 편의성**: 전체 평균 **C등급** (78%). 공급자(기관) 중심 설계로 개발자 경험이 열악하다.
- **AI 시대 적합성**: 전체 **62%가 E등급**. OpenAPI 스펙 부재, HTTP 200 일관 반환, XML 우선, CORS 미지원으로 AI agent 연동이 구조적으로 어렵다.
- **data.go.kr 과대평가**: "1개 ID로 70% 커버"는 오해. 실제 게이트웨이(Pattern A)는 18%뿐.
- **포털 파편화**: 169개 포털에 130개 별도 ID. 동일 데이터가 2~3곳에서 다른 키/포맷으로 제공.
- **개선 우선순위**: OpenAPI 스펙 의무화 → HTTP 상태코드 정상화 → JSON 기본 전환 → 통합 API Gateway.

공공 데이터의 가치는 **얼마나 많이 공개하느냐**가 아니라 **얼마나 쉽게 활용할 수 있느냐**로 결정된다. AI 시대에 공공 데이터가 진정한 국가 자산이 되려면, 공급자 중심에서 사용자 중심으로의 패러다임 전환이 필요하다.
